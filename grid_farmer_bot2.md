# Bithumb 그리드/농부/회복 터틀 전략 문서

이 문서는 현재 `bithumb_grid_farmer` 코드 기준의 전략 동작을 한국어로 정리한 운영 설계서입니다. 과거 아이디어 문서가 아니라, 실제 봇의 상태 흐름과 대시보드/텔레그램에서 보이는 값을 이해하기 위한 기준 문서입니다.

## 1. 목표

이 봇의 목적은 BTC/KRW를 대상으로 하락 구간에서는 정해진 그리드로 분할 매수하고, 더 깊은 하락 구간에서는 농부 전략으로 추가 매수를 검토한 뒤, 반등 구간에서는 회복 터틀 로직으로 통합 포지션을 관리하는 것입니다.

핵심 원칙:

- 매수와 매도 판단은 현재 상태 파일을 기준으로 일관되게 처리한다.
- 실거래 주문은 반드시 안전장치와 한도 검사를 통과해야 한다.
- 로그는 사후 검증이 가능하도록 사이클, 차수, 주문 ID, 손익, 매도 이유를 남긴다.
- 대시보드와 텔레그램은 같은 상태 파일과 로그를 읽되, 현재 상태와 기간 로그의 의미를 구분한다.

## 2. 전체 상태 흐름

```text
GRID
  ├─ 그리드 레이어 매수/매도
  ├─ 모든 레이어 OPEN 후 농부 신호 감시
  └─ 농부 확정 매수 발생 시 회복 포지션 관리

FARMING / HOLDING
  ├─ 농부 추가 매수 조건 확인
  ├─ 통합 회복 포지션 손익 계산
  └─ 회복 터틀 청산 신호 확인

COOLDOWN
  └─ 청산 이후 재진입 대기
```

실제 타입상 phase는 `GRID`, `FARMING`, `HOLDING`, `COOLDOWN`을 지원합니다. 현재 운영에서는 그리드 단계 안에서 농부 신호를 감시하고, 매수 체결 이후 회복 포지션 관리를 이어가는 방식으로 동작합니다.

## 3. 그리드 전략

### 3.1 사이클과 레이어

그리드 사이클은 `cycleId`로 구분됩니다. `data/bot_state.json`에는 현재 사이클의 기준 가격, 레이어 목록, 각 레이어의 상태가 저장됩니다.

| 상태 | 의미 |
| --- | --- |
| `WAITING` | 아직 매수되지 않은 차수 |
| `OPEN` | 매수 체결 후 보유 중인 차수 |
| `SOLD` | 매수 후 매도까지 완료된 차수 |

각 레이어는 `buyPrice`, `sellPrice`, `amountKrw`, `qty`, `buyCount`, `sellCount`, `buyOrderId`, `sellOrderId`를 가집니다.

### 3.2 1차 진입 방식

그리드 오픈 포지션이 0개일 때 1차 매수 기준은 설정으로 선택할 수 있습니다.

| 모드 | 설명 |
| --- | --- |
| `CURRENT_PRICE` | 현재가를 기준으로 1차 매수를 시작 |
| `N_MULTIPLE` | 현재가에서 `N * 사용자 배수`만큼 하락한 가격을 1차 매수 기준으로 사용 |

`N`은 확정 일봉의 변동폭을 바탕으로 계산합니다. KST 기준 오늘 일봉은 아직 확정되지 않았으므로 제외합니다. 따라서 N 값은 보통 하루에 한 번 바뀌며, 장중 가격 변동만으로 계속 바뀌지 않습니다.

### 3.3 레이어 가격 고정

한 사이클에서 만들어진 레이어 가격은 고정됩니다. 예를 들어 5차가 매수된 뒤 매도되어 `SOLD`가 되었고, 가격이 다시 5차 매수가까지 내려오면 이전 5차 매수가와 같은 가격에서 다시 매수를 시도합니다.

레이어 가격이 바뀌는 경우:

- 그리드 리셋
- 새로운 사이클 시작
- 설정 변경 후 재초기화
- 계좌 자본 자동 반영에 따른 레이어 재구성
- 터틀 청산 이후 재진입 구조 변경

### 3.4 트레일링 매도

그리드 매도는 기본 목표 수익 또는 트레일링 되돌림 조건으로 실행될 수 있습니다. 최근 로그에는 매도 이유가 `metadata`로 남습니다.

| 필드 | 의미 |
| --- | --- |
| `sellReason` | `TAKE_PROFIT` 또는 `TRAILING_PULLBACK` |
| `peakPrice` | 보유 중 추적된 최고가 |
| `peakReturnPct` | 최고가 기준 수익률 |
| `exitReturnPct` | 매도 시점 수익률 |
| `returnPullbackPct` | 최고 수익률에서 되돌린 폭 |
| `trailingPullbackPct` | 설정된 되돌림 기준 |
| `trailingStopPrice` | 트레일링 스탑 가격 |
| `targetSellPrice` | 기본 목표 매도가 |

대시보드는 새 로그에 이 값이 있으면 매도 이유를 더 구체적으로 표시합니다. 과거 로그처럼 메타데이터가 없는 경우에는 `BITHUMB_WS`, `BITHUMB_REST` 같은 기존 reason이 보일 수 있습니다.

## 4. 농부 전략

### 4.1 농부 기준 매수가

농부 매수 전의 **농부 기준 매수가**는 마지막 그리드 차수의 매수가입니다. 즉, 보유 중인 그리드 중 가장 최근에 산 가격이 아니라, 현재 사이클에서 가장 깊은 그리드 레이어의 `buyPrice`를 기준으로 합니다.

농부 매수 후에는 기준이 바뀝니다.

| 상태 | 대시보드 기준값 |
| --- | --- |
| 농부 매수 전 | 마지막 그리드 차수의 매수가 |
| 농부 1회 이상 매수 후 | 직전 농부 매수가 |

이 구조는 그리드의 마지막 방어선 아래에서 농부 진입을 시작하기 위한 것입니다.

### 4.2 농부 매수 실행 조건

농부는 단순히 가격만 내려왔다고 바로 사지 않습니다. 가격 도달 조건과 확인 필터를 함께 봅니다.

주요 필터:

- 농부 진입가 도달 여부
- 농부 단계별 쿨다운
- 3일 하락률 제한
- 변동성 폭발 여부
- 장기 추세 필터
- MA5 추세 필터
- 거래대금 비율과 절대 거래대금
- 종가 위치
- 양봉 조건
- 2일 연속 양봉 조건
- 급락/투매 조건
- 매수 후 현금 방어 조건

`ENABLE_FARMER_CONFIRMED_BUY=false`이면 농부 신호만 기록하고 실제 주문은 하지 않습니다.

### 4.3 농부 포지션

농부 포지션은 `farmerPositions`에 단계별 leg로 저장됩니다. 각 leg는 단계, 수량, 원가, 주문 ID, 매수 시각을 가집니다.

농부 매수는 페이퍼 모드에서는 가상 체결로 기록되고, 실거래 모드에서는 Bithumb 주문 실행기를 통해 실제 주문으로 처리됩니다.

## 5. 회복 터틀 전략

회복 터틀은 농부 매수 이후 포지션을 복구하는 청산 전략입니다. 그리드 포지션과 농부 포지션을 합친 통합 회복 포지션을 기준으로 판단합니다.

### 5.1 통합 포지션

통합 포지션 계산에 포함되는 항목:

- `OPEN` 상태의 그리드 레이어 수량과 원가
- 농부 포지션 수량과 원가

계산 결과는 예상 시장가치, 예상 순손익, 예상 수익률로 이어집니다.

### 5.2 청산 조건

회복 터틀은 예상 순손익이 양수일 때만 실제 청산 후보로 봅니다. 손실 상태에서는 신호가 있더라도 청산을 막는 것이 기본 의도입니다.

| 신호 | 의미 |
| --- | --- |
| `2N_TRAIL` | 최고가 대비 2N 기준 트레일링 이탈 |
| `MA5_EXIT` | MA5 기준 이탈 |
| `N_DAY_LOW_BREAK` | N일 저가 이탈 |
| `TAKE_PROFIT_1` | 1차 부분 익절 |
| `TAKE_PROFIT_2` | 2차 부분 익절 |

`ENABLE_RECOVERY_TURTLE_SELL=false`가 기본값입니다. 이 상태에서는 신호와 예상 손익만 기록하고 실제 매도는 실행하지 않습니다.

### 5.3 분할 매도

회복 포지션이 큰 경우 한 번에 모두 팔지 않고 지정 금액 단위로 나누어 팔 수 있습니다.

| 설정 | 의미 |
| --- | --- |
| `RECOVERY_USE_SLICE_ORDER` | 분할 주문 사용 여부 |
| `RECOVERY_TURTLE_SLICE_ORDER_KRW` | 1회 분할 매도 금액 |
| `RECOVERY_TURTLE_SLICE_INTERVAL_SECONDS` | 분할 주문 간격 |

## 6. 주문 실행 방식

### 6.1 페이퍼 주문

기본 모드는 페이퍼 주문입니다. 이 경우 Bithumb에 실제 주문을 내지 않고, 현재 가격 기준으로 체결된 것처럼 상태와 로그를 갱신합니다. 페이퍼 주문 ID는 보통 `paper-buy-...`, `paper-sell-...` 형태입니다.

### 6.2 실거래 주문

실거래 주문은 다음 조건을 모두 만족해야 합니다.

- `ENABLE_REAL_ORDERS=true`
- `REAL_ORDERS_CONFIRM=I_UNDERSTAND_REAL_BITHUMB_ORDERS`
- Bithumb API 키 존재
- 1회 주문 한도와 총 운용 한도 통과
- `GRID_BOT_MOCK_PRICE` 미설정

주문은 기본적으로 공격형 지정가입니다. 매수는 현재가보다 약간 높은 지정가, 매도는 현재가보다 약간 낮은 지정가를 사용해 체결 가능성을 높입니다.

체결 확인 흐름:

1. 지정가 주문 제출
2. `GRID_BOT_AGGRESSIVE_LIMIT_WAIT_MS` 동안 체결 조회
3. 미체결이면 취소 시도
4. 취소 응답과 재조회 결과에서 지연 체결 여부 확인
5. 최종 미체결이면 오류 로그 기록 후 다음 루프에서 재판단

따라서 텔레그램에서 "limit order was not filled" 오류를 받았더라도, 이후 다른 주문 ID로 재시도되어 정상 체결될 수 있습니다.

## 7. 가격 이벤트 처리

현재 기본 가격 입력은 Bithumb WebSocket ticker입니다. 가격 이벤트가 들어오면 봇 루프가 깨어나고, 일정 시간 이벤트가 없으면 안전 점검 주기로 상태를 다시 확인합니다.

| 값 | 의미 |
| --- | --- |
| `BITHUMB_WS` | 웹소켓 가격 이벤트를 계기로 판단 |
| `BITHUMB_REST` | REST 조회 또는 주문 체결 확인 결과 |

`BITHUMB_WS`는 매도 이유가 아니라 가격 신호 출처입니다. 실제 매도 이유는 새 로그의 `metadata.sellReason`을 우선 확인합니다.

## 8. 대시보드 표시 기준

대시보드는 `data/bot_state.json`과 `data/trading_logs/btc_master_log.jsonl`을 읽어 표시합니다.

| 화면 | 기준 |
| --- | --- |
| Grid Status | 현재 `bot_state.json`의 레이어 상태 |
| 최근 실거래 로그 | 현재 cycleId 기준 최신 거래 로그 |
| Daily Report | 기간 내 거래 로그와 현재 사이클 통계를 분리 표시 |
| 농부 기준 매수가 | 농부 매수 전에는 마지막 그리드 차수 매수가, 매수 후에는 직전 농부 매수가 |

Daily Report의 `Buy fills`는 기간 전체 로그 기준입니다. 현재 Status의 `OPEN`은 현재 상태 기준입니다. 사이클 리셋이 있으면 두 값은 다를 수 있습니다.

## 9. 로그 설계

거래 로그는 JSON Lines 형식으로 `data/trading_logs/btc_master_log.jsonl`에 저장됩니다.

| action | 의미 |
| --- | --- |
| `GRID_BUY` | 그리드 매수 체결 |
| `GRID_SELL` | 그리드 매도 체결 |
| `FARMER_SIGNAL` | 농부 신호 기록 |
| `FARMER_BUY` | 농부 매수 체결 |
| `RECOVERY_EXIT_SIGNAL` | 회복 터틀 매도 신호 |
| `RECOVERY_SELL` | 회복 터틀 매도 체결 |
| `PHASE_CHANGE` | 단계 전환 |
| `BOT_ERROR` | 오류 기록 |

`GRID_SELL`에는 가능한 경우 최고 수익률, 매도 시점 수익률, 되돌림 폭, 트레일링 가격을 `metadata`에 남깁니다.

## 10. 운영 검증 기준

실거래 운영 중 이상 여부를 볼 때는 다음 순서로 확인합니다.

1. `pm2 list`에서 네 프로세스가 online인지 확인
2. `pm2 env bithumb-grid-bot-paper`에서 실거래 환경 변수가 반영됐는지 확인
3. `data/bot_state.json`에서 현재 cycleId와 레이어 상태 확인
4. `btc_master_log.jsonl`에서 해당 주문 ID와 action 확인
5. Bithumb 실제 체결 내역과 로그의 orderId 비교

가짜 OPEN 레이어가 생긴 경우에는 보정 매수를 하지 않고, 해당 레이어를 `WAITING`으로 되돌리는 방향이 더 안전합니다. 실제 매수 없이 상태만 OPEN이 된 경우 그대로 두면 다음 매도 판단이 실제 잔고와 어긋날 수 있습니다.

## 11. 디스크와 PM2 관리

AWS 루트 디스크가 작으면 PM2 로그와 백테스트 산출물이 빠르게 쌓입니다.

| 우선순위 | 대상 | 비고 |
| --- | --- | --- |
| 1 | PM2 로그 | `pm2-logrotate` 권장 |
| 2 | `data/backtests` | 재생성 가능 |
| 3 | 오래된 빌드 산출물 | 필요 시 재빌드 가능 |
| 마지막 | `data/trading_logs` | 거래 기록이므로 임의 삭제 비권장 |

운영 서버에서는 EBS를 20GB 이상으로 확장하는 것을 권장합니다.

## 12. 남은 개선 후보

- 대시보드에서 실거래 주문 재시도 흐름을 더 명확하게 표시
- 체결 지연, 취소 실패, 재조회 체결을 별도 이벤트로 세분화
- 회복 터틀 실매도 활성화 전 별도 리허설 모드 추가
- 로그 보존 정책과 압축 정책 자동화
- Daily Report의 기간, 사이클, 현재 상태 비교 표 강화
