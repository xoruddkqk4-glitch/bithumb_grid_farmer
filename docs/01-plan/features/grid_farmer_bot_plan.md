## 1. 배경 및 목적 (Why)

비트코인 시장의 극심한 변동성과 유동성 리스크를 수학적으로 통제하는
하이브리드 알고리즘 트레이딩 시스템을 설계한다.

- 평시: 20차수 그리드로 변동성 수확
- 폭락 시: 다회차 농부매매(평가금 2배 물타기)로 평단 방어
- 상승 전환: 통합 매도(2N 트레일링 OR MA5 하회)로 추세 통째 수확
- 자금 관리: 현재 총자본 기반 역산 동적 배분으로 입출금 리스크 흡수
- 운영: Next.js + Supabase 대시보드 + 기간별 엑셀(XLSX) 정산 다운로드

**핵심 철학:** "살 때는 정교하게 쪼개어 사고, 팔 때는 추세를 통째로 먹는다."

## 2. 자금 설계 (확정 - 핵심 변경사항)

### 2.1. 그리드 비율 하향 조정 (17.2% → 15.8%)

원본 명세의 17.2%는 농부 3차(평가금×2)까지 자금이 닫히지 않아(약 3.3% 초과)
**그리드 비율을 15.8%로 하향 조정**한다. (안전마진 5% 포함)

```
농부 3차까지 총 투입 ≈ 그리드 비율 × 6.008 (복리 증폭 계수)
- 17.2% × 6.008 = 103.3%  → ❌ 자금 고갈 3.3% 초과
- 16.64% × 6.008 = 100%   → ⚠️ 한계 (마진 0)
- 15.8% × 6.008 = 94.9%   → ✅ 안전마진 5% 확보 (채택)
```

### 2.2. 자본 규모별 매입금 테이블 (15.8% 기준, 20차 운용, 간격 -1%)

```
1회 매입금 = 총 자본금 × 0.158 / 20 = 총 자본금 × 0.0079
```

| 총 자본금 | 그리드 총액(15.8%) | **1회 매입금(20차)** |
|:--:|:--:|:--:|
| 1,000만 원 | 158만 원 | **79,000원** |
| 2,000만 원 | 316만 원 | **158,000원** |
| 3,000만 원 | 474만 원 | **237,000원** |
| 4,000만 원 | 632만 원 | **316,000원** |
| 5,000만 원 | 790만 원 | **395,000원** |

> 매 사이클 시작 시(또는 청산 후 재진입 시) API로 총자본을 재스캔하여
> 위 비율대로 1회 매입금을 동적으로 재산정한다.

### 2.3. 농부매매 베팅 규칙 (확정)

- **정의:** "포지션 2배" = **진입 시점 보유 평가금만큼 추가 투입** (A안)
- 농부 1차: 그리드 20차 종료 직후, 09:05 필터 충족 시 평가금만큼 추가 투입
- 농부 2차: 1차 체결가 대비 -15% 하회 시, 09:05 필터 충족 시 평가금만큼 추가 투입
- 농부 3차: 2차 체결가 대비 -15% 하회 시, 09:05 필터 충족 시 평가금만큼 추가 투입
- 농부 1·2·3차 모두 동일 규칙 (현금 한도 초과 시 잔여 현금 전액으로 cap)

### 2.4. 동적 입출금 대응

| 단계 | 입출금 발생 시 처리 |
|:--|:--|
| 그리드 중 | 이미 체결된 차수는 불변, 남은 차수만 새 자본 기준 재산정 |
| 농부 중 | 베팅이 평가금에 종속되므로 자동 반영, 현금 한도(고갈)에만 영향 |
| 정책 | 사이클 진행 중 출금 시 "농부 방어력 하락" 경고 표시 |

## 3. 매매 로직 (확정)

### 3.1. 매수 (24시간, trailing_stop_bot)

```
그리드 매수: 최초 진입가 대비 -1% 간격 20분할, 24시간 실시간 체결
농부 매수  : 가격 도달은 24시간 감시, 실제 체결은 09:05 3대 필터 통과 시 (해석 A)
```

**농부 3대 필터 (09:05, 전날 확정 일봉 기준, AND 결합):**
1. 전날 종가 > MA5 (5일 이평선 안착)
2. MA5_today >= MA5_yesterday (기울기 우상향)
3. 전날 거래량 > 최근 5일 평균 거래량 × 150%

### 3.2. 매도 / 통합 청산 (확정)

```
청산 조건 = (실시간 수익률 > 0)  AND  (조건① 2N 이탈  OR  조건② MA5 하회)
```

| 조건 | 발동 | 담당 봇 | 게이트 |
|:--|:--|:--|:--|
| ① 2N 이탈 | 24시간 실시간 | trailing_stop_bot | 체결 직전 실시간 수익률 > 0 |
| ② MA5 하회 | 09:05 1회 | ma5_trend_bot | 체결 직전 실시간 수익률 > 0 |

- 두 조건은 **OR** (먼저 감지한 쪽이 전량 시장가 청산)
- 게이트(수익률 > 0)는 **두 조건 모두 체결 직전 실시간 기준으로 통일**
- 중복 청산 방지: 청산 직전 보유 평가금 < 10,000원이면 조용히 종료

### 3.3. 무손절 원칙 (전략적 결정)

```
MA5 하회 + 수익률 음수 → 청산하지 않고 홀딩 (추세 장기 수확)
```

- 본 시스템은 의도적으로 손절매를 두지 않음
- 농부 3차 올인 후 추가 하락 시 현금 0 상태로 무기한 홀딩 가능
- ⚠️ 리스크 고지는 §7 참조

## 5. 시스템 아키텍처 (확정)

### 5.1. 프로세스 분리 (백엔드 봇)

| 봇 | 구동 | 핵심 임무 | N(ATR) 역할 |
|:--|:--|:--|:--|
| `trailing_stop_bot.py` | 24시간 (PM2) | 그리드/농부 매수, 2N 트레일링 매도 | 읽기 전용 사용 |
| `ma5_trend_bot.py` | 09:05 1회 (crontab `5 9 * * *`) | 농부 필터 검증, MA5 하회 청산, N 계산 | 계산 및 저장 |

### 5.2. N(ATR) 계산 - 터틀 원전 방식

```
True Range (TR) = max(
  당일 고가 - 당일 저가,
  |당일 고가 - 전일 종가|,
  |당일 저가 - 전일 종가|
)
N = (19 × 전일 N + 당일 TR) / 20   (Wilder 20일 평활)
```

- 일봉 기준 → 09:05 ma5_trend_bot이 하루 1회 계산
- **계산 방식 (확정):** 매일 업비트 API로 직전 20일 일봉을 조회하여 N(ATR) 산출
- **첫 구동 처리 (확정):** 별도 시드 로직 불필요. 첫 구동 시에도 동일하게
  직전 20일 일봉을 조회해 즉시 계산 (매일 동일 로직 적용)
- trailing_stop_bot은 저장된 N값을 24시간 읽기 전용으로 사용 (2N 트레일링)

### 5.3. 상태 공유 (bot_state.json)

```json
{
  "phase": "GRID | FARMING | HOLDING | COOLDOWN",
  "last_exit_time": "ISO8601",
  "cooldown_until": "ISO8601",
  "N_value": 0,
  "farmer_stage": 0,
  "highest_price": 0,
  "grid_entry_price": null
}
```

- **원자적 쓰기:** 임시파일 작성 후 rename(os.replace)으로 읽기 충돌 방지
- ma5_trend_bot이 write, trailing_stop_bot이 주로 read

### 5.4. 사이클 상태머신

```
GRID → FARMING → HOLDING → (청산) → COOLDOWN(5분) → GRID(재진입) → ...
```

### 5.5. 동시성 정책 (확정 - 매수 일시정지 불필요)

별도의 "09:05 매수 일시정지" 장치는 **불필요**하다. 상태머신이 이를 자연히 보장한다.

```
- 청산은 HOLDING 상태에서만 발생 → 이미 모든 매수가 끝난 상태
- 청산 직후 COOLDOWN(5분) 진입 → 이 동안 매수 동작 없음
- 따라서 "청산 도중 매수 끼어들기"가 발생할 여지가 구조적으로 없음
```

**전제 조건(필수):** `trailing_stop_bot`은 매 루프에서 phase를 확인하여
`HOLDING` / `COOLDOWN` 상태일 때는 어떤 매수도 시도하지 않는다.
(매수 동작은 `GRID` / `FARMING` 상태에서만 허용)

## 6. 데이터 로깅 · 정산 · 대시보드 (확정)

자산 흐름의 안정적 보관과 사용자의 직관적 정산 관리를 위해
**'기록 저장'**과 **'파일 변환/조회'**의 역할을 이원화한다.

### 6.1. 데이터 저장 계층 (2단계 전환 전략)

| 단계 | 저장소 | 용도 |
|:--|:--|:--|
| 1단계 (MVP) | 로컬 JSON `./data/trading_logs/btc_master_log.json` | 봇이 모든 매매 데이터를 시간순 append |
| 2단계 (확장) | **Supabase (PostgreSQL)** | Next.js 빌드업 시 JSON을 DB로 일대일 대체 적재 |

**로그 레코드 스키마(안):**
```json
{
  "timestamp": "ISO8601",
  "cycle_id": "사이클 식별자",
  "action": "GRID_BUY | FARMER_BUY | LIQUIDATE",
  "stage": "그리드 차수 or 농부 차수",
  "price": 0,
  "qty": 0,
  "amount_krw": 0,
  "avg_price": 0,
  "pnl_pct": 0,
  "reason": "2N_TRAIL | MA5_EXIT | null"
}
```

### 6.2. 기간별 엑셀(XLSX) 다운로드 (확정)

- 사용자가 대시보드에서 **원하는 기간(단일 날짜 또는 시작~종료일)**을 지정
- 시스템이 마스터 로그/DB에서 해당 기간 데이터만 필터링
- 실시간으로 독립된 정산 보고서 파일 즉석 생성·제공
  - 예: `btc_report_2026-06-01_2026-06-03.xlsx`
- 포함 항목: 매매 내역, 일별 손익, 누적 수익률, 사이클별 요약

### 6.3. 대시보드 (Next.js + Supabase)

| 영역 | 기능 |
|:--|:--|
| 실시간 현황 | 현재 phase, 보유 포지션, 평단가, 실시간 수익률, N값 |
| 사이클 히스토리 | 과거 사이클별 진입/청산/수익률 타임라인 |
| 자금 모니터 | 총자본, 예수금, 농부 차수별 투입 현황, 방어력 게이지 |
| 정산/다운로드 | 기간 선택 → XLSX 다운로드 |
| 알림 | 출금 시 방어력 하락 경고, 청산 발생 알림 |

**기술 스택:**
- 프론트엔드: Next.js (App Router)
- 백엔드/DB: Supabase (PostgreSQL + Auth + Realtime)
- 봇 ↔ 대시보드 연동: 봇이 Supabase에 매매 로그 적재 → 대시보드가 조회/구독

## 7. 리스크 고지 (필수 명문화)

본 시스템은 **무손절 추세추종 전략**이다. 투입 자금은 다음 리스크를 감수한다.

- 최대 자금 투입 지점: 농부 3차 (최초 진입가 대비 약 -41.5%)
- 농부 3차 이후 추가 하락 시: 현금 소진 상태로 무기한 홀딩
- BTC 역사적 하락 사례: -77%(2018), -75%(2022)
- → 투입 자금은 반드시 "장기 미사용 가능 여유자금"이어야 함
- → 슬리피지로 인해 "수익률 양수일 때만 매도" 원칙이 미세하게 깨질 수 있음 (비용 수용)
